The Problem

A benefits-management system with 9 independent feature teams (enrollment, validation, assessment, notifications, admin…) needed to ship independently without redeploying the entire app — and each micro-frontend had to resolve its own asset paths at runtime across different environments.

My Approach

I built the shell host using Angular Module Federation with a critical runtime fix: the public path for each MFE must be read from window.__env at call-time, not at module-initialization time — because the env config file hadn't finished executing yet at startup, leaving every entry undefined and causing MFEs to fall back to the shell URL.

The Code

app-routing.module.ts

function loadRemoteModuleWithPublicPath(remoteName: string, 
exposedModule: string) { 
  const moduleName = formatModuleNameFromPath(remoteName); 
 
  // Read window.__env at CALL time, not module-init time. 
  // The static map was evaluated once at routing-module load; 
  // if env.js hadn't finished executing, every entry was undefined, 
  // causing MFEs to fall back to the shell URL. 
  const commonConfig = window.__env?.COMMON_CONFIG; 
  const publicPath = commonConfig?.[moduleName] ?? commonConfig?.UI; 
 
  (window as any).__INJECTED_PUBLIC_PATH_BY_RUNTIME__ = publicPath; 
  return loadRemoteModule({ type: 'manifest', remoteName, exposedModule }); 
}

webpack.config.js

module.exports = withModuleFederationPlugin({ 
  name: 'abcd_ui', 
  shared: { 
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }), 
    '@ngx-translate/core': { singleton: false },          // deliberate: each MFE owns its i18n 
    '@designlibrary/abc-library': { singleton: true, strictVersion: false } 
  }, 
});

The Problem

An eligibility system used by government workers needs: auto-logout on idle, cross-tab session sync (logout in tab A = logout in tab B), debounced loading spinners (no flashing on fast calls), and 401 auto-redirect — all without blocking Angular's change detection.

My Approach — 4 interlocking pieces

1 · Idle timer outside Angular's zone — session timeout.service.ts

private startIdleTimer(): void { 
  this.ngZone.runOutsideAngular(() => {        // timer doesn't trigger CD 
    this.idleInterval = setInterval(() => { 
      this.idleTime++; 
      if (this.idleTime >= this.IDLE_TIMEOUT) { 
        this.ngZone.run(() => this.onTimeout.next()); // re-enter only when needed 
      } 
    }, 1000); 
  }); 
}

2 · Cross-tab logout via StorageEvent — cross-tab sync.service.ts

private setupStorageListener(): void { 
  globalThis.addEventListener('storage', (event: StorageEvent) => { 
    if (event.key === 'app-logout-event' && event.newValue) { 
      this.logoutDetected$.next(); 
    } 
  }); 
} 
 
public broadcastEvent(key: string, value: string): void { 
  localStorage.setItem(key, value); 
  setTimeout(() => localStorage.removeItem(key), 100);  // fire-and-forget 
}

3 · Debounced spinner interceptor — loading.interceptor.ts

intercept(req, next) { 
  if (this.activeRequests === 0) { 
    this.spinnerTimeout = setTimeout( 
      () => this.store.dispatch(setShowSpinner({ showSpinner: true })), 
      this.SPINNER_DELAY            // config-driven — no flash on fast calls 
    ); 
  } 
  this.activeRequests++; 
  return next.handle(req).pipe( 
    finalize(() => { 
      this.activeRequests--; 
      if (this.activeRequests === 0) { 
        clearTimeout(this.spinnerTimeout); 
        this.store.dispatch(setShowSpinner({ showSpinner: false })); 
      } 
    }) 
  ); 
}

4 · 401 auto-logout — security.interceptor.ts

intercept(req, next) { 
  return next.handle(req).pipe( 
    catchError((error: HttpErrorResponse) => { 
      if (error.status === 401) this.logoutService.logout(); 
      return throwError(() => error); 
    }) 
  ); 
}

The Result

  • No change-detection cycles from idle timers (runOutsideAngular).
  • Logout in any tab propagates to all open tabs within 100ms.
  • Spinner only appears when a request exceeds SPINNER_DELAY — no flicker.
  • Any 401 cleanly tears down the session and redirects to login.

The Idea

A recursive resolveSectionsDeep() routine lazily resolves deeply nested Formly form configurations from JSON files using forkJoin + switchMap recursion. Entire dashboard pages become config driven with zero hardcoded HTML — new screens ship by adding a JSON file, not by writing components.

Why It Matters

  • Non-developers can compose new pages from configuration.

How I Structured a Module Federation Shell for 9+ Micro-Frontends

When several teams ship into the same product, a single monolithic Angular app becomes the bottleneck: one team's broken build blocks everyone, and every release is an all-or-nothing event. The fix was a Module Federation shell — a thin host application that orchestrates independently deployable remotes.

Manifest-driven, not build-time

The most important decision was resolving remotes at runtime from a manifest, rather than hard-coding their URLs at build time. This means a remote can be redeployed to a new URL without rebuilding the shell — the host simply reads the latest manifest.

// mf.manifest.json — remotes resolved at runtime, not build time 
{ 
  "enrollment": "https://cdn/enrollment/remoteEntry.js", 
  "validation": "https://cdn/validation/remoteEntry.js", 
  "assessment": "https://cdn/assessment/remoteEntry.js" 
}

Each remote is loaded lazily, on navigation, straight from the manifest entry:

loadRemoteModule({ 
  type: 'module', 
  remoteEntry: manifest[name], 
  exposedModule: './Module' 
}).then(m => m.RemoteModule);

Singleton shared dependencies

Angular, RxJS, and NgRx must be singletons across the shell and every remote. Loading two copies of Angular leads to subtle, maddening bugs — DI tokens that don't match, two router instances, broken change detection. The Module Federation config marks core libraries as singleton: true with a requiredVersion, so the first one wins and everyone shares it.

What this bought us

  • Independent deploys: each of the 9+ modules (e-Application, Enrollment, Validation, Assessment, Notifications, Admin…) ships on its own pipeline.
  • Fault isolation: a remote that fails to load degrades one route, not the whole app.
  • Zero-downtime updates: paired with a version-polling service, the shell detects a new remote and reloads cleanly.
The shell should own as little as possible: routing, shared state wiring, and the manifest. Everything else belongs to the remotes.

The result is an architecture where teams move at their own pace — and the product still feels like one coherent app to the user.

Centralizing State Across Micro-Frontends with NgRx

In a micro-frontend system, state is the hard part. Each remote needs its own slice, but they also share cross-cutting concerns — the authenticated user, feature flags, notifications. I standardized this with a shared NgRx store registered in the shell, plus feature stores contributed by each remote.

Feature stores per remote

Each remote registers its own feature state with provideState() and its own effects with provideEffects(). Because the store is a singleton (see the Module Federation post), these slices all live in one tree — fully visible in the Redux DevTools, with time-travel debugging across the whole app.

The logout problem

The subtle bug: after logout, feature slices kept their data. The next user to log in on a shared machine could briefly see the previous session's state. The clean fix is a single meta-reducer that resets the root state on a logout action — every feature store rehydrates from its initial state automatically. 

// Wipe every feature slice on logout — one meta-reducer, all remotes 
export function clearStateOnLogout(reducer) { 
  return (state, action) => { 
    if (action.type === AuthActions.logout.type) { 
      state = undefined; // reset root, feature stores re-init clean 
    } 
    return reducer(state, action); 
  }; 
}

Principles that kept it maintainable

  • Selectors are the public API of a slice — components never reach into raw state.
  • Effects own all side effects (HTTP, storage), keeping reducers pure and testable.
  • One source of truth for auth/user, consumed by every remote through shared selectors.

Standardizing this flow measurably reduced state-related bugs — because every remote followed the same, debuggable pattern instead of inventing its own.

Session Security Done Right: Idle Timeout, JWT Refresh & Cross-Tab Logout

Session management looks trivial until you account for reality: users keep the app open in five tabs, leave for an hour, and expect to be logged out everywhere at once. I built this as four cooperating layers.

1. Silent JWT refresh

Access tokens are short-lived. A timer refreshes them on an interval well before expiry, so the user never sees an interruption. Crucially, this timer runs outside Angular's zone — otherwise every tick triggers change detection across the whole app for no visual reason.

this.zone.runOutsideAngular(() => { 
  interval(5 * 60_000).subscribe(() => this.refreshToken()); 
});

2. Idle timeout

A 60-minute inactivity timer (reset on user interaction) logs the session out automatically — protecting unattended machines.

3. The 401 interceptor

If a request ever comes back 401 — token revoked server-side, refresh failed — an HTTP interceptor catches it, clears the session, and routes to login. This is the safety net behind the proactive refresh.

4. Cross-tab logout sync

The piece teams most often miss. When the user logs out in one tab, every other tab must follow. The storage event makes this trivial — writing a key to localStorage fires the event in all other tabs of the same origin. 

// Broadcast logout to every open tab 
logout() { 
  localStorage.setItem('logout-event', Date.now().toString()); 
  this.clearSession(); 
} 
 
window.addEventListener('storage', (e) => { 
  if (e.key === 'logout-event') this.clearSession(); 
});

Why layers?

No single mechanism is sufficient. Proactive refresh keeps sessions smooth; idle timeout protects walk-aways; the 401 interceptor is the server-authoritative backstop; cross-tab sync keeps the user's mental model intact. Together they make the session feel invisible when it should — and airtight when it matters.
